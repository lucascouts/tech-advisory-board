# Architecture

This document is for maintainers and contributors. It describes how the
Technical Advisory Board plugin is put together — the mental model, the
pipeline, the persistence layer, the hook web, the invariants that must
not be broken, and the extension points.

If you are trying to *use* the plugin, read the `README.md` first. If you
are trying to *parse* a synthesis file, read `docs/SYNTHESIS_CONTRACT.md`.
This document assumes you want to change something.

## 1. Mental model

Think of a single TAB session as one meeting of an expert panel. The
actors are:

- **Moderator** — the skill body itself. It orchestrates the session,
  reads references on demand, spawns subagents, writes state, emits the
  synthesis. Runs in the main conversation context.
- **Scout** — the Moderator wearing a different hat during Phase 2 to
  produce the shortlist and discard table. Not a separate subagent.
- **Champions** — `tech-advisory-board:champion`. Advocacy subagents.
  Each defends one stack. Parallel fan-out.
- **Advisors** — `tech-advisory-board:advisor`. Dimensional evaluators.
  Each scores every champion proposal from one angle (cost, DX,
  scalability, …). Parallel fan-out. Independent — they never see each
  other's outputs.
- **Researcher** — `tech-advisory-board:researcher`. Ecosystem data
  collector (versions, CVEs, community health). Has `memory: project`
  so it accumulates across sessions, which requires a COI disclosure.
- **Auditor** — `tech-advisory-board:auditor`. Adversarial reviewer of
  the near-final synthesis. Mandatory in Complete / Complete+ /
  Rechallenge. Has `memory: project` and preloads the
  `tab:verification-checklist` skill.
- **Supervisor** — `tech-advisory-board:supervisor`. Conditional
  consensus-theater gate. Fires only when §12.1 triggers match
  (≥80% advisors score the same proposal ≥8/10, std-dev <1.5, no
  scores <4, no direct challenges). Does not re-score; detects
  structural bias.

The board runs a 13-phase pipeline that is scaled down or up by
**mode**. Modes are chosen by a complexity classifier; the user can
override.

| Complexity | Mode | Champions | Advisors | Cross-exam | Auditor |
|---|---|---|---|---|---|
| Trivial | Instant | 0 | 0 | — | — |
| Simple | Fast | 2 | 2 | — | — |
| Moderate | Standard | 3 | 3-4 | 1 round | conditional |
| High | Complete | 3-4 | 4-5 | multi | **required** |
| Very High+ | Complete+ | 4-5 | 5-6 | multi | **required** |

See `skills/tab/references/flow-and-modes.md` for the authoritative
table and the ASCII flow diagram.

## 2. Pipeline

```
   Phase -1   Bootstrap             init-dir, load-config, resume detection
      │
   Phase 0    Context extraction    team, budget, constraints
      │
   Phase 1    Baseline research     N Researchers, parallel
      │
   Phase 2    Landscape scan        Scout hat: shortlist + discard table
   Phase 2.5  Triage reinforcement  discard reversals, premise check,
      │                             steel-man of best discard
   Phase 3    Champion debate       N Champions, parallel
      │                             (Vanguard subtype when applicable)
   Phase 4    Cross-examination     Champion-to-champion
      │                             Option A (subagents) or B (main ctx)
   Phase 5    Advisor evaluation    N Advisors, parallel, independent
   Phase 5.5  Supervisor gate       if §12.1 consensus-theater fires
   Phase 6    Score consolidation   matrix, divergences, wildcard
   Phase 6.5  Auditor               mandatory in Complete+ / Rechallenge
      │
   Phase 7    Synthesis             synthesis.json + report.md
                                    new-adr in Standard+ modes
   Stop-gate  validate-synthesis-json.sh + validate-claims.sh
```

Every phase transition is persisted to `state.json` (hot) and
`state-full.json` (full claims registry). See `references/persistence-
protocol.md` for the exact shapes.

## 3. Persistence layer

### 3.1 `.tab/` — per-project state

```
.tab/
├── config.json              merged user + project config snapshot
├── index.md                 ADR index (regenerated by bin/new-adr)
├── decisions/
│   ├── .lock                flock sentinel (see bin/new-adr)
│   └── NNNN-<slug>.md       MADR ADRs
└── sessions/
    ├── archived/            idle sessions moved here by init-dir
    └── <YYYY-MM-DD-slug>/
        ├── state.json                  hot: phase, next_phase
        ├── state-full.json             full: claims_registry, subagents
        ├── telemetry.json              totals, mcp_failures, elicitations,
        │                                teammate_idles
        ├── synthesis.json              canonical output (see
        │                                 docs/SYNTHESIS_CONTRACT.md)
        ├── report.md                   human-readable render
        ├── timeline-events.ndjson      monitor.sh event stream (rotated)
        ├── timeline.html               self-contained viewer
        ├── denials.ndjson              permission-denial audit trail (rotated)
        ├── stalls.ndjson               stall-guard detections (rotated)
        └── crash.json                  last N StopFailure records
```

### 3.2 `${CLAUDE_PLUGIN_DATA}` — cross-project state

```
${CLAUDE_PLUGIN_DATA}/
├── vanguard-timeline.json   cross-project Vanguard maturity ledger
└── shared-cache/
    └── cache.json           research cache keyed by sha256(query+tool+date_window)
```

These files survive plugin updates (per `/en/plugins-reference#persistent-
data-directory`). Both writers (`bin/vanguard-timeline append`,
`bin/cache invalidate|prune`) serialize through
`scripts/lib/json_atomic.atomic_update()` so concurrent sessions from
different projects don't race.

### 3.3 Schema contracts

All persistence formats have JSON Schema contracts under `schemas/`
(Draft 2020-12). The important ones:

| Schema | Scope | Contract stance |
|---|---|---|
| `synthesis.schema.json` | public | versioned (`schema_version: 1`), major-bump on breaks |
| `config.schema.json`    | public | project-owners may hand-edit `.tab/config.json` |
| others                  | internal | may change between plugin versions without warning |

### 3.4 Atomic state helper

`scripts/lib/json_atomic.py` provides the read-modify-write primitives
that every state mutator uses:

- `atomic_update(path, mutator, default, timeout_seconds)` for JSON
  objects. Acquires an exclusive `fcntl.flock` on `<path>.lock`, reads
  the target (or `default` if missing/corrupt), calls `mutator`, writes
  to `<path>.tmp`, atomically renames via `os.replace()`, releases the
  lock.
- `atomic_append_ndjson(path, record, max_lines, max_bytes,
  max_rotations)` for line-oriented files. Same lock protocol, plus
  automatic rotation when either cap is hit.
- `acquire_flock(fd, timeout_seconds)` as a primitive for scripts that
  don't fit the read-modify-write shape (e.g. `bin/new-adr` where the
  critical section is "scan + reserve NNNN + write file + update index").

Any new state mutator MUST go through one of these. Direct `open(path,
"w")` on files in `.tab/` or `${CLAUDE_PLUGIN_DATA}/` is a regression.

## 4. Hooks

Hook handlers are registered in two layers:

- **Global layer** — `hooks/hooks.json` at the plugin root. Active
  whenever the plugin is enabled.
- **Skill-scoped layer** — `hooks:` block in the YAML frontmatter of
  `skills/tab/SKILL.md` and `skills/rechallenge/SKILL.md`. Active only
  during the lifecycle of that skill, per
  `/en/hooks#hooks-in-skills-and-agents`. Used for `PreCompact`,
  `PostCompact`, `Stop` so non-TAB sessions never dispatch them.

They fall into two categories regardless of layer:

### 4.1 Gates — can influence host decisions

- **`PreCompact`** (skill-scoped; `flush-state.sh`) — vetoes compaction
  with `{"decision":"block","reason":...}` when the session is
  mid-Phase 4 (cross-exam) or mid-Phase 6.5 (auditor). Watchdog-capped
  at 2s so a stuck script never blocks the host.
- **`Stop`** (skill-scoped; `validate-on-stop.sh`) — blocks session
  close with `{"decision":"block","reason":...}` when
  `validate-synthesis-json.sh` or `validate-claims.sh` fail. The
  Moderator sees the feedback and must re-emit a valid synthesis.
- **`PermissionRequest`** (global; `on-permission-request.sh`, scoped
  via `if:` to `Read(**/.tab/**)`, `Glob(**/.tab/**)`,
  `Grep(**/.tab/**)`) — auto-approves read-only access to TAB's own
  artifacts so the Moderator can re-read `state.json`,
  `synthesis.json`, etc. without interactive friction.

### 4.2 Instrumentation — observational only

- **`SessionStart`** — resume hint, runtime-deps warnings (node
  missing, Agent Teams env missing).
- **`UserPromptSubmit`** — auto-inject project context when the prompt
  looks like a TAB intent.
- **`PostCompact`** (skill-scoped) — re-hydrate the Moderator with
  phase, mode, language lock, claims digest.
- **`PostToolUse`** — telemetry for `.tab/sessions/**` writes; plugin
  version bump when `plugin.json` is edited.
- **`PostToolUseFailure`** — classify MCP errors, append to
  `telemetry.mcp_failures[]`, emit fallback hint after 3×.
- **`TaskCreated` / `SubagentStart` / `SubagentStop` / `TaskCompleted`**
  — subagent lifecycle timing into `state-full.subagents_invoked[]`.
- **`StopFailure`** — capture API-error session ends into `crash.json`.
- **`CwdChanged` / `FileChanged`** — detect workspace switches and
  external mutations.
- **`InstructionsLoaded`** — re-inject the locked `session_language`
  after session start or compaction.
- **`Elicitation` / `ElicitationResult`** — audit every
  `AskUserQuestion` round into `telemetry.elicitations[]`.
- **`PermissionDenied`** — append to `denials.ndjson`.
- **`Notification`** — route rechallenge failures to the `channel` MCP
  server.
- **`TeammateIdle`** — capture Agent Teams stalls into
  `telemetry.teammate_idles[]`.
- **`SessionEnd`** — idle-session archival safety net.
- **`ConfigChange`** — record user/project/local/policy/skills settings
  mutations into `state-full.config_changes[]`; flags sessions whose
  `config_snapshot` diverges from the current config mid-flight.
- **Worktree lifecycle** — observed via three complementary sources,
  all appending to `<session>/worktrees.ndjson` with a `source` field
  identifying provenance:
  - **`PostToolUse(EnterWorktree|ExitWorktree)`**
    (`scripts/on-worktree-tool.sh`) — primary observer for the
    Rechallenge skill. Pairs `enter` and `exit` events by the
    host-provided `tool_use_id`; captures `duration_ms` when the host
    attaches it. Source: `tool`.
  - **`WorktreeRemove`** (`scripts/on-worktree-remove.sh`) —
    observational counterpart for worktrees removed by the harness
    (e.g. cleanup of `isolation: worktree` sub-agents). Source:
    `remove`.
  - **`SubagentStart` / `SubagentStop` snapshot**
    (`scripts/snapshot-worktrees.sh`) — safety-net diff against
    `git worktree list --porcelain`. Catches creations / removals that
    bypassed the two hooks above, such as agents calling
    `git worktree add` via Bash or harness-native paths that skip
    the tool surface. The first run on a fresh session establishes a
    silent baseline so pre-existing worktrees aren't logged. Source:
    `snapshot`.

  > **Why not register a `WorktreeCreate` hook?** The Claude Code
  > harness treats that event as *productive*: the hook's stdout is
  > consumed as the worktree path, and registering any handler
  > disables the native `git worktree add` code path. A
  > best-effort instrumentation hook that prints nothing would
  > therefore break `isolation: worktree` for every plugin in every
  > project where TAB is enabled. See the contract table in §4.4.

### 4.3 Monitors

Declared in `monitors/monitors.json`, run as long-lived background
processes (per `/en/plugins-reference#monitors`). Both monitors use
`when: on-skill-invoke:tab` so they arm only after the first `tab`
dispatch of a session — non-TAB sessions pay zero I/O:

- **`tab-session-monitor`** (`scripts/monitor.sh`) — streams NDJSON
  events for phase transitions, subagent start/return, budget
  thresholds. Mirrors to `<session>/timeline-events.ndjson` (rotated
  at 10k lines / 5 MB / 3 rotations).
- **`tab-stall-guard`** (`scripts/stall-guard.sh`) — second-line
  defense after the Claude Code native 10-min stall kill. Detects
  subagents stuck >30 min (pre-first-token, MCP setup hangs) and emits
  `stall-detected` events.

> A session that invokes only `rechallenge` (without first invoking
> `tab`) runs without these monitors. This is intentional — rechallenge
> is short-lived and does not spawn the fan-out that `tab` does.

### 4.4 Host hook contract — observational vs productive

Not every hook event in Claude Code is purely observational. Some are
**productive**: the harness consumes the hook's stdout as part of its
own control flow, and registering any handler disables the native code
path that would otherwise run. Treating a productive event as a
best-effort instrumentation point silently breaks the host behavior
behind it.

Read this table before adding a new hook registration — confirm the
event's role in the current Claude Code release.

| Event | Role | What the harness consumes | Safe to register as observational? |
|---|---|---|---|
| `PreToolUse`, `Stop`, `PreCompact`, `PermissionRequest` | **Gate** | `decision: "block" / "approve" / null` + `reason` | Yes, **as long as the script never emits a `decision`** — silent stdout is treated as no-op. |
| `WorktreeCreate` | **Productive** | Hook stdout = absolute path of the created worktree. The presence of any handler disables `git worktree add` | **No.** A handler that doesn't actually create the worktree and print its path will fail every `isolation: worktree` sub-agent globally. |
| `SessionStart` `additionalContext` | Productive (additive) | Stdout appended to the model's context | Safe to leave stdout empty; only write when you mean to inject context. |
| `WorktreeRemove`, `PostToolUse`, `SubagentStart`, `SubagentStop`, `TaskCreated`, `TaskCompleted`, `CwdChanged`, `FileChanged`, `ConfigChange`, `Elicitation`, `Notification`, `TeammateIdle`, `SessionEnd`, `InstructionsLoaded`, `PermissionDenied`, `StopFailure` | **Observational** | Nothing — exit code is the only signal | Yes. Best-effort scripts that exit 0 are fine. |

When in doubt, search the host binary (`strings $(which claude)`) for
the event name and look at the surrounding error templates — productive
events typically carry a "hook succeeded but returned no …" message
nearby. The `WorktreeCreate` lesson (logged in `CHANGELOG` 0.1.3) is the
canonical example.

## 5. Critical invariants

These properties must hold; protect them when you change code.

### 5.1 Session language is locked

The Moderator detects language from the first user message and
persists it in `state.json.language` + `state-full.session_language`.
Every downstream output (subagent prompts, synthesis, report,
elicitations) uses this value. `InstructionsLoaded` hook re-injects
it after compaction. Never re-detect mid-session.

### 5.2 Confidence tags are monotone

Every quantitative or factual claim carries `high-conf` / `med-conf` /
`low-conf` / `unverified`. When two subagents disagree on the
confidence for the same claim, record the **lower**. Consumers rely on
this to drive gating decisions.

### 5.3 Two-source rule

Every `high-conf` claim must cite ≥2 independent sources. The Researcher
enforces this at collection time; the Auditor checks it in Complete+.
`validate-claims.sh` asserts it at the Stop gate.

### 5.4 COI disclosure for memoized agents

Agents with `memory: project` (Researcher, Auditor) load prior-session
notes into context. They MUST emit a COI disclosure at the top of
their output declaring stacks previously touched and mitigations
(identity rotation). Absence of the disclosure is a Stop-gate failure
in Complete+ via `conflicts_of_interest[]` required-fields check.

### 5.5 Advisor independence

An Advisor NEVER sees another Advisor's output. The Moderator
dispatches them in parallel via `subagent_type: "tech-advisory-
board:advisor"` with identical prompts modulo the dimension. Merging
advisor outputs is Moderator work; the Advisor's prompt explicitly
states "you have not seen any other advisor's evaluation".

### 5.6 Discard fairness

The landscape discard table must apply criteria uniformly. If a tool
is discarded for reason X, every other tool that would fail X is also
discarded — never cherry-picked. The Auditor's `discard-fairness`
dimension checks this.

### 5.7 Auditor is mandatory in Complete+ / Rechallenge

Hard-coded in `validate-synthesis-json.sh`. Mode-dependent. Disabling
it requires `auditor_enabled: false` in `userConfig` AND an explicit
`auditor_override_reason` in `.tab/config.json` — both, not either.

### 5.8 ADR numbering is atomic

`bin/new-adr` holds an exclusive flock on `.tab/decisions/.lock` for
the whole "scan NNNNs → pick next → write file → update index" cycle.
The ADR write uses `O_EXCL` as a second safety belt. Two concurrent
`new-adr` processes always produce distinct NNNNs.

### 5.9 `schema_version` is a hard contract

Breaking changes to `synthesis.json` bump `schema_version`. Consumers
fail soft on unknown versions. Never break v1 without landing v2.

## 6. Models by role

Default assignment (`references/model-policy.md`):

| Role | Default model | Effort |
|---|---|---|
| Champion | `claude-opus-4-7` | `max` |
| Advisor | `claude-sonnet-4-6` | `max` |
| Researcher | `claude-sonnet-4-6` | `max` |
| Auditor | `claude-opus-4-7` | `xhigh` |
| Supervisor | `claude-sonnet-4-6` | `max` |

Overrides:

- Per-role via `userConfig.{champion,advisor,auditor}_model`.
- Per-mode via `userConfig.model_policy_overrides`
  (`{"Instant":"haiku","Fast":"sonnet","Complete":"opus"}` style).
- Policy via `userConfig.model_policy`:
  - `static`: honor *_model knobs only.
  - `budget-aware` (default): downgrade Champion / Auditor toward
    Sonnet when <30% of the cost ceiling remains.
  - `context-aware`: also reacts to cache-hit rate, MCP availability,
    novelty vs. vanguard-timeline, prompt-cache warmth.

`scripts/model-policy.sh` is the resolver. Called by the Moderator
before every subagent spawn.

## 7. Graceful degradation

The plugin runs on a spectrum from full-featured to heavily degraded.
Each external dependency fails in a well-defined way.

| Missing | Effect |
|---|---|
| `python3` | Most hook scripts no-op (exit 0); validators skip; Stop-gate becomes a no-op. Session continues with lower-confidence claims. |
| `node` | LSP servers + some MCP bridges unavailable; Read/Grep fallback. Warning on SessionStart. |
| `git` | `extract-context.sh` emits partial data; Rechallenge falls back to in-place. |
| `jq` | Not used at runtime; only examples/ci-github-actions.yml uses it. |
| MCP servers (perplexity, context7, brave-search) | Researcher / Champion / Auditor fall back to built-in WebSearch; claims tagged `[unverified]`. |
| `CLAUDE_CODE_EXPERIMENTAL_AGENT_TEAMS` (unset) | `agent_team_mode=agent_teams` hard-fails; `agent_team_mode=auto` silently falls back to subagents; `agent_team_mode=subagents` (default) is unaffected. |
| `tab:verification-checklist` skill missing | Auditor runs with its internal checklist; depth reduced. |
| `CLAUDE_CODE_REMOTE=true` | Monitors (`monitor.sh`, `stall-guard.sh`) no-op; statusline no-op; session still runs. |
| `CLAUDE_CODE_SUBPROCESS_ENV_SCRUB=1` | `channel` MCP server runs in degraded mode (no Telegram/webhook transport env); pushes become silent no-ops. |

## 8. Extension points

### 8.1 New archetype

Add to `skills/tab/references/archetypes.md`. Include:
- Name, one-line description
- Academic credentials profile for identity-card generation
- Default bias, default blind spot
- When the Moderator should assign this archetype

Nothing else to change — the Moderator picks archetypes from this file
at Phase 3.

### 8.2 New advisor dimension

Add to `skills/tab/references/specialists.md`. Include:
- Dimension name
- Scoring calibration (what does 1-10 mean on this dimension?)
- What this dimension should and should not consider

The Moderator can also generate domain-specific advisors at session
time; the reference file covers the Core roster only.

### 8.3 New tool in a subagent's allowlist

Edit the corresponding `agents/<name>.md` frontmatter `tools:` list.
Plugin validator (`claude plugin validate .`) enforces the constraint
that plugin agents cannot declare `mcpServers`, `hooks`, or
`permissionMode`. `tools:` and `disallowedTools:` are both supported.

### 8.4 New hook

1. Add the handler script to `scripts/`.
2. Register in `hooks/hooks.json` with the right event name (see
   `/en/hooks` for the full enum) and optional `matcher` / `if`
   filters.
3. If the hook mutates `.tab/` state, route through
   `scripts/lib/json_atomic` — do not open files directly.
4. Document in `skills/tab/references/hooks-catalog.md`.

### 8.5 New bin command

1. Add the script to `bin/` (shebang, `chmod +x`).
2. No registration needed — Claude Code auto-adds `bin/` to the Bash
   tool's PATH when the plugin is active.
3. Document in `README.md` bin table.
4. If it mutates shared state, use `scripts/lib/json_atomic`.

### 8.6 New userConfig knob

1. Add under `plugin.json.userConfig` with description, type, default,
   `sensitive` flag.
2. Document in `README.md` user-config table.
3. Read it from the environment as
   `CLAUDE_PLUGIN_OPTION_<knob_name>` in any hook/bin script.
4. The Moderator reads it from `state.json.config_snapshot` (snapshotted
   at Phase -1 so changes mid-session don't apply until the next
   session).

should-not-trigger, near-miss, intent-detection, and failure-mode

`feat(telemetry)`.

## 10. Where to look when something breaks

- Plugin fails to load — `claude plugin validate .`
- Subagent not dispatching — check `agents/<name>.md` frontmatter
  parses as valid YAML, `tools:` list
- Hook not firing — run `claude --debug` and grep for the event name
- Synthesis rejected at Stop — read
  `<session>/crash.json` and the `validate-synthesis-json.sh` output
- Budget exceeded — `bin/compute-cost --from-telemetry
  <session>/telemetry.json`
- Stall suspected — `<session>/stalls.ndjson` (monitor-mediated) or
  `<session>/state-full.json` (look for `ended_at: null` entries)
- Cache issues — `bin/cache stats` and `bin/cache list`
- Vanguard assessment out of date — `bin/vanguard-timeline list`
